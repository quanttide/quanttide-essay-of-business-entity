# 数字资产契约

用契约驱动 Git 仓库结构治理。

在团队协作中，Git 仓库的文件夹结构往往因人而异，容易陷入混乱。引入「数字资产契约」的概念，可以将仓库的组织规范以机器可读的形式固定下来，像代码一样进行版本管理和自动化校验，从而确保知识库或代码库的整洁与一致性。

## 概念定义

数字资产契约将隐性的工作流转化为显性的、可版本控制的、可迁移的数字资产契约。

## 核心价值

- 多仓分离：文档、应用、资产分离管理
- 子模块化：各资产独立版本控制
- 契约驱动：以 `.quanttide/` 目录作为契约事实源

## 什么是数字资产契约

数字资产契约是一个 YAML 或 JSON 文件，定义了仓库应遵循的目录结构规则。

- 根目录必须包含 README.md 和 docs/ 文件夹
- 禁止出现临时文件（如 *.tmp）或特定目录（如 node_modules/）
- 文档目录下的文件名需采用短横线命名（kebab-case）
- 源码目录的嵌套深度不得超过 3 层

典型规则字段包括 mustContain（必须存在）、mustNotContain（禁止存在）、namingConvention（命名规范）、allowedExtensions（允许的扩展名）和 maxDepth（最大深度）。

## 如何落地实施

推荐使用开源工具 repo-structure，它专为校验 Git 仓库结构设计，支持 pre-commit 钩子和 CI 集成。

1. 编写契约文件（如 .digital-asset-contract.yaml），纳入版本控制
2. 配置本地钩子：在 .pre-commit-config.yaml 中添加 repo-structure，每次提交前自动校验
3. 设置 CI 门禁：在 GitHub Actions 等 CI 流程中运行 repo-structure validate，阻止不符合契约的 PR 合并
4. 契约演进：修改契约需经团队评审，保持规范的生命力

## 价值与扩展

数字资产契约将仓库结构从「潜规则」变为「明规则」，让新人能快速上手，自动化工具确保持续合规。

未来可进一步扩展为「数字资产契约家族」，例如增加文件内容契约（校验 Markdown 的 Front Matter）、命名规范契约（统一文件名风格）、依赖契约（确保文件成对出现）等，形成覆盖数字资产全生命周期的治理体系。

通过契约驱动，团队能够以工程化的方式管理知识库，让结构像代码一样可靠。
