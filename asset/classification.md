# 分类

## 概述

数字资产的分类规范以**范畴论**为理论基础定义"类别"（Category），以**类型论**为理论基础定义"类型"（Type）。这一分类精准地捕捉了数字资产在**业务语义**与**结构约束**两个维度的本质区别。

基于这一分类规范，可以构建一个既具备**数学严谨性**，又能在异构平台间实现**互操作性**的统一资产模型（UAM, Unified Asset Model）。

---

## 核心概念

### 类别 (Category) —— 业务语义的态射网络

**理论基础**：范畴论关注的是"对象"与"态射"。资产作为"对象"，其**类别**定义了它在业务域中的位置以及与其他资产的关系（态射）。

**作用**：解决"这个资产在业务中是什么角色？"的问题。

**示例**：合同、设计稿、源代码发布包、客户工单。

**特性**：**关系性**。同一类别下的资产遵循相同的业务组合逻辑（如积范畴、余积范畴）。

### 类型 (Type) —— 结构约束的类型构造

**理论基础**：类型论关注的是如何通过规则构造新类型，防止非法状态的发生（命题即类型，程序即证明）。

**作用**：解决"这个资产由什么构成？如何被验证和消费？"的问题。

**示例**：Git仓库、S3对象、Kubernetes CRD、JSON Schema。

**特性**：**构造性与可验证性**。类型定义了资产的数据契约、接口签名和不变式。

## 统一资产模型架构

### 模型要素

```
[统一资产模型]
├── 资产实体 (Asset Entity)
│   ├── 唯一标识 (UID): 跨平台的全局ID
│   ├── 类别 (Category): 指向 Category 对象的引用 (态射源)
│   ├── 类型 (Type): 指向 Type 对象的引用 (类型构造器)
│   ├── 版本 (Version): 语义化版本或哈希
│   └── 状态 (State): 基于类型论推导出的合法状态机
│
├── 类别空间 (Category Space) —— 基于范畴论
│   ├── 对象 (Objects): 业务概念 (如 PRD, 设计, 代码)
│   ├── 态射 (Morphisms): 业务流转 (如 评审通过, 部署上线)
│   ├── 函子 (Functors): 跨平台的映射规则 (如 Figma 资产 -> 前端代码资产)
│   └── 交换图 (Commutative Diagrams): 确保业务逻辑在不同平台的一致性
│
└── 类型空间 (Type Space) —— 基于类型论
    ├── 原始类型 (Primitive Types): Git, S3, 数据库记录
    ├── 复合类型 (Composite Types): 结构体、枚举、联合类型
    ├── 依赖类型 (Dependent Types): 类型依赖于值 (如 "端口号必须在 1024-65535 之间")
    └── 接口契约 (Interface): 定义如何读取、写入、执行该资产
```

### 关键机制

#### 类别的组合性（范畴论的运用）

跨平台的统一，本质上是在不同平台的资产之间建立**函子**关系。

**场景**：在 Jira（平台A）有一个"任务"（类别：需求），在 GitHub（平台B）有一个"PR"（类别：代码变更），在 Figma（平台C）有一个"Frame"（类别：UI设计）。

**模型实现**：

- 定义一个**积范畴** \( \text{需求} \times \text{设计} \times \text{代码} \)
- 通过**极限**（Limit）构造一个"特性交付"对象。任何平台上的资产变更，都会在这个统一的极限对象上触发态射（即业务流转）
- 这确保了不同平台间的资产不是孤立的，而是构成一个完整的业务**交换图**

#### 类型的约束性（类型论的运用）

跨平台统一的最大挑战是**平台特性（异构性）**。类型系统负责将平台特性抽象为类型构造器。

**场景**：Git 仓库的类型定义。

- 在 GitHub 上它表现为 `Repository` 对象
- 在 GitLab 上它表现为 `Project` 对象

**模型实现**：

- 定义一个抽象类型 `GitRepoType`
- 通过**依存类型**定义该类型必须满足的约束：`必须有 main 分支`、`必须关联 deploy key`
- 当接入 GitHub 平台时，实例化为 `GitHubRepo`；接入 GitLab 时，实例化为 `GitLabProject`。只要满足类型签名，上层业务（类别）就不需要关心底层差异

## 跨平台统一实现

以统一"代码仓库"资产在"GitHub"和"GitLab"上的表示为例：

| 维度 | 模型定义 | GitHub 实例 | GitLab 实例 | 跨平台统一逻辑 |
|:---|:---|:---|:---|:---|
| **类别** | `代码资产` | `代码资产` | `代码资产` | **态射**：无论是 GH 还是 GL，都映射到同一个业务态射"代码审查"。范畴论保证了业务流的同构性。 |
| **类型** | `GitRepo` | `GitHubRepo` | `GitLabProject` | **类型构造**：抽象类型定义了 `cloneUrl`、`defaultBranch`。具体类型实现了平台特定的 API 调用逻辑。 |
| **资产ID** | `repo:sha256:<内容哈希>` | `github:owner/repo` | `gitlab:owner/project` | **类型归属**：ID 前缀区分平台，但内容哈希确保即便迁移平台，业务类别不变。 |
| **操作** | `PullRequest` (态射) | `PR` 对象 | `MR` 对象 | **函子映射**：模型定义 `MergeRequest` 函子，将 GitHub 的 PR 态射与 GitLab 的 MR 态射映射到统一的"代码合并"态射上。 |

## 与 Platform Engineering 和 IDP 的契合

### Platform Engineering 与 IDP 的本质诉求

**Platform Engineering** 的目标是将基础设施与底层平台的复杂性抽象为开发者可自服务的产品，而 **IDP（Internal Developer Platform，内部开发者平台）** 是这个产品的具体形态。

核心诉求包括：

- **抽象异构性**：屏蔽底层多云、多 SaaS、多自建系统的差异
- **自服务能力**：开发者通过声明式配置即可完成资源申请、环境搭建、部署发布
- **可组合性**：平台组件能像乐高积木一样灵活组装
- **可验证性**：平台能自动校验配置的合法性、依赖的完整性
- **可观测性与可追溯性**：所有资产变更有迹可循

### 范畴论支撑 IDP 的"可组合、可追溯"

在 IDP 中，资产通过业务关系构成**工作流网络**。范畴论将业务流建模为范畴：

- **态射的组合性**：两个首尾相接的态射可以合成新态射，平台可自动推导跨阶段的组合，无需手动编排
- **交换图的一致性**：在异构平台间（如 Jira → GitHub → ArgoCD），无论走哪条路径，最终结果一致
- **函子实现平台抽象**：接入新平台只需实现一个函子，将平台特定概念映射到统一业务范畴
- **极限实现资产聚合**：形式化构造复合资产（如"微服务 = 代码仓库 + 镜像仓库 + 配置 + 数据库"）

### 类型论支撑 IDP 的"可验证、可信任"

类型论为 IDP 提供严格的资产契约与验证能力：

- **依赖类型捕获业务约束**：如"端口号必须在 1024-65535 之间"、"只有审批通过后才能部署到生产"
- **类型检查实现"左移"**：在部署前捕获配置错误，保证所有必填字段存在、依赖关系满足、不变式成立
- **命题即类型**：资产类型 T 是命题"存在满足约束的资产"，配置是该命题的证明。类型检查通过即命题被证明
- **类型安全的重构**：资产类型变化时，依赖配置被标记为不兼容，防止遗漏

### 操作系统的类比

| 传统操作系统概念 | 基于范畴论+类型论的 IDP 概念 |
|------------------|------------------------------|
| 进程 | 资产实例 |
| 系统调用 | 态射（业务操作） |
| 文件类型 | 资产类型 |
| 权限检查 | 类型约束 + 依赖类型 |
| 可执行文件 | 通过类型检查的资产配置 |
| Shell 脚本 | 工作流（由态射和类型推导） |
| 内核 | 平台引擎（维护范畴与类型系统） |

## 模型优势

### 可推理性

由于引入了**类型论**，资产的合法性可以在接入时就通过类型检查（Type Checking）完成，避免了运行时因平台差异导致的配置错误。引入**范畴论**，业务的完整性和流转路径可以通过交换图进行形式化验证。

### 平台解耦

业务逻辑（**类别**）与物理存储/平台 API（**类型**）分离。新增平台（如从 GitLab 迁移到 Bitbucket）时，只需实现新的**类型实例**（满足 `GitRepo` 类型契约），而无需修改上层**类别**（如"持续交付流水线"）的逻辑。

### 资产追溯

通过范畴论中的**泛性质**，可以构建资产的谱系。例如，从"生产环境部署资产"可以唯一回溯到"源代码资产"和"配置资产"，形成不可篡改的资产血缘图。

## 总结

基于**范畴论（类别）**与**类型论（类型）**构建的统一资产模型，本质上是将**业务语义的逻辑结构**与**技术实现的数据结构**进行了正交分解。

- **类别**提供了高维度的业务视图，让我们能够以数学方式描述资产间的流转、组合和映射，解决了"跨平台业务一致性"问题
- **类型**提供了底层的契约保障，通过强类型的接口隔离了异构平台的实现差异，解决了"跨平台技术异构性"问题

**范畴论**赋予 IDP 结构化的业务逻辑建模能力，使其成为**可组合、可追溯、可推导**的系统。**类型论**赋予 IDP 严格的资产契约与验证能力，使其成为**可验证、可信任、自服务**的系统。

二者结合，IDP 不再仅仅是一个工具集合或 API 聚合，而是一个**形式化的、可被程序信任的"操作系统"**——开发者与平台之间通过数学语言进行沟通，平台的每一个行为都可以从基础理论推导出来，最终实现 Platform Engineering 的理想：**让开发者专注于业务价值，而平台以绝对正确的方式承载一切**。
